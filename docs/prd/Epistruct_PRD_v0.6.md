# Epistruct (에피스트럭트) — Product Requirements Document (PRD)

> 버전: v0.4
> 작성일: 2026-06-01
> 상태: Draft — 아키텍처 결정사항 반영 중
> 이름 유래: Epistemology(인식론/지식론) + Structure(구조). 인간이 지식을 인지·학습하는 과정을 시스템적 구조(Graph)로 변환해 저장한다는 목표의 물리적 결합.
> 기반: 사용자가 파일 기반(thinker 에이전트)으로 실제 운용·검증한 지식 모델을 멀티유저 서비스로 제품화한다.
> 한 줄 정의: AI 시대에 인간이 지식을 직접 인지·이해하고, 원리 단위로 통찰을 확장하도록 돕는 개인화 지식 구조화 플랫폼.

---

## 1. 배경과 문제 정의

### 1.1 문제

AI(특히 LLM)의 보편화로 다음과 같은 역설적 현상이 발생하고 있다.

- **사고력 약화**: 답을 즉시 얻을 수 있어 직접 사고하는 시간이 줄어든다.
- **판단력 부재**: 정보가 맞는지 틀린지 검증할 기준과 역량이 약해진다.
- **정보 과부하**: AI가 생성하는 정보의 속도와 양이 한 사람이 인지·소화할 수 있는 한계를 넘는다.

### 1.2 기존 솔루션의 한계 — "사람이 루프에서 빠진다"

현재 유행하는 "LLM Wiki" 패턴(예: Obsidian + Claude Code 기반 자동 지식베이스)의 진짜 문제는 입력 자동화 자체가 아니라, **인간이 지식 생성·관리 루프에서 배제된다**는 데 있다.

- **AI가 넣고 AI가 꺼내 쓴다**: 수집·구조화·검색을 전부 AI가 수행한다. 사람은 결과만 받는다.
- **망가졌을 때 개입이 어렵다**: 구조가 잘못 형성되거나 중복·오류가 누적돼도, 사람이 어디를 어떻게 고쳐야 할지 손댈 지점을 잃는다.
- **전체를 파악할 수 없다**: 자료가 늘수록 "AI는 찾을 수 있지만 사람은 인지할 수 없는" 블랙박스가 된다. 내 지식인데 내가 모른다.

즉, 결과물은 쌓이는데 **그것이 내 머릿속 구조와 일치하지 않고, 통제권도 사라진다.** 이것이 핵심 결함이다.

### 1.3 목표 — 사람이 인지·통제하는 지식 구조 + 원리 전이

Epistruct가 만들려는 것은 다음과 같다.

1. **압축**: 같은 의미의 정보를 요약·통합해 부피를 줄인다.
2. **연관 가시화**: 개념 간 관계를 사람이 한눈에 이해할 수 있게 보여준다.
3. **중복 제거**: 유사·중복 노드를 감지해 정리한다.
4. **사람의 인지·통제**: 사용자가 자기 지식 전체를 파악하고, 직접 개입·수정할 수 있다. AI는 보조자이지 소유자가 아니다.

그리고 그 위에 한 단계 더 나아간 목표가 있다.

> **원리 전이 (Principle Transfer)**
>
> 지식의 표면적 사실이 아니라 **원리(Principle)의 핵심**을 추출하고, 서로 다른 도메인의 원리끼리 연결·전이시켜 색다른 이해와 통찰을 얻게 한다.
>
> 예) "도파민의 보상 예측 오차" ↔ "강화학습의 TD 오차" / "네트워크 백프레셔" ↔ "재고 관리의 수요 조절". 표면 주제는 다르지만 원리는 같다. 이 연결이 통찰을 만든다.

이 "원리 전이"가 단순 지식 저장 도구와 Epistruct를 가르는 핵심 차별점이며, 5장의 지식 모델(P/C/M/D)로 구현된다.

### 1.4 핵심 철학

> **AI가 지식을 대신 소유·생성하는 것이 아니라, 사용자가 지식을 인지·구조화하는 과정을 AI가 돕는다.**
> AI의 역할은 사서(librarian)가 아니라 **학습 코치 + 지식 번역가 + 원리 연결자**다.

## 2. 목표와 비목표

### 2.1 목표 (Goals)

- 사용자가 넣은 정보를 개인의 학습 스타일·관심 도메인에 맞게 구조화한다.
- 지식을 압축하고, 연관관계를 가시화하고, 중복을 제거해 **사람이 인지·통제 가능한** 구조로 만든다.
- 원리 단위로 핵심을 추출하고, 도메인 간 원리를 전이시켜 통찰을 만든다.
- 저장에 그치지 않고 맥락에 맞는 지식을 능동적으로 꺼내준다.
- 사용자 간 지식을, 받는 쪽의 이해·정리를 거쳐 서로의 구조를 해치지 않고 융합할 수 있게 한다.
- iOS / Android / Web에서 동일하게 동작하는 멀티유저 서비스로 제공한다.

### 2.2 비목표 (Non-Goals)

- AI가 사용자를 루프에서 배제한 채 "정답 요약"을 자동 생산하는 것.
- 다른 스코프의 지식을 사용자의 이해 없이 그대로 복사·참조하게 하는 것.
- 초기 단계에서의 실시간 협업 편집(구글 독스 수준).
- 전문 임상·의료·법률 자문 제공.
- 초기 단계에서의 자체 LLM 학습/파인튜닝(외부 API 활용으로 시작).

## 3. 타겟 사용자

### 3.1 핵심 페르소나

**P1. 깊이를 잃고 싶지 않은 학습자**
- AI에 의존하면서 직접 사고하는 시간이 줄었다고 느낀다.
- 자료는 쌓이는데 자기 머릿속 구조와 일치하지 않아 통제력을 잃었다고 느낀다.
- 사실 암기가 아니라 원리 수준의 이해와 통찰을 원한다.

**P2. 지식 공유·협업 학습자**
- 스터디·팀 단위로 지식을 나누고 싶다.
- 단, 남의 정리를 그대로 받으면 자기 맥락이 깨져 불편함을 느낀다.

### 3.2 학습 다양성 전제

같은 주제라도 사용자마다 관심과 정리 방식이 다르다. 이를 시스템 1급 개념으로 다룬다.

- **관심 도메인(What)**: 예) SQL → 문법 / 튜닝 / 설계 / 운영 중 어디에 집중하는가.
- **학습 스타일(How)**: 개념형(정의→원리→예시), 사례형(문제→해결→일반화), 마인드맵형(중심→방사), 질문형(소크라테스식).

## 5. 지식 모델 (Knowledge Model)

> 본 모델은 사용자가 파일 기반(thinker 에이전트)으로 실제 운용해 검증한 구조를 계승한다. 노드 타입·관계·진입 기준은 그대로 따르고, 멀티유저를 위해 **스코프**를 직교 차원으로 추가한다.

### 5.1 노드 타입 (4계층)

| Type | 접두사 | 정의 | 예시 |
|---|---|---|---|
| Principle | `P:` | 도메인 독립적 보편 원리 | `P:time-space-tradeoff` |
| Concept | `C:` | 원리가 특정 영역에서 구체화된 형태 | `C:caching` |
| Manifestation | `M:` | 구체적 기술/도구/현상 | `M:it/infrastructure/redis` |
| Domain | `D:` | 지식 영역(분류 축) | `D:computer-science` |

### 5.2 관계 (6종)

| 관계 | 방향 | 의미 |
|---|---|---|
| `DECOMPOSE_TO` | C → P | 이 개념은 이 원리로 분해된다 |
| `MANIFESTS_AS` | P → C | 이 원리가 이 개념으로 나타난다 |
| `INSTANTIATED_BY` | C → M | 이 개념이 구체적 형태로 나타난다 |
| `CONNECTS_TO` | C → C | 직접적 관계(의존·보완·확장) |
| `ANALOGOUS_TO` | C → C | 구조적 유사성(다른 도메인) |
| `BELONGS_TO` | C → D | 이 개념은 이 영역에 속한다 |

### 5.3 진입 기준 (= 검증 알고리즘)

노드 승격/강등 판정은 시스템의 핵심 알고리즘으로 구현된다(난이도 요건 충족).

- **P 진입**: 3개 이상 분야에서 등장, 더 단순한 원리로 분해 불가, 트레이드오프 또는 불변 법칙을 기술.
- **C 진입**: 1~2개 P로 분해, 2개 이상 서로 다른 기술/도구에서 등장, 기존 C의 스케일 변형이 아님, 직관적 이름.
- **미충족 시**: 기존 노드에 흡수(example/참조 추가) 또는 강등(C→M, P→C). `--review`류 점검을 백그라운드 작업으로 자동화.

### 5.4 데이터 계층 모델 (Source → Chunk → Node)

지식 생성 파이프라인은 3개의 독립 레이어로 구성된다. 레이어는 각각 다른 목적을 가지며, 상위 레이어가 삭제돼도 하위 레이어는 독립적으로 유지된다.

```
Layer 1: Source (원본)     사용자가 넣은 것. 사람이 읽기 위해 보존.
    ↓ 청킹(chunking)
Layer 2: Chunk (가공본)    AI가 처리하는 단위. 임베딩 저장. RAG 기반.
    ↓ LLM 추출 + 사용자 확정
Layer 3: Node (지식)       P/C/M/D. 사용자가 확정한 구조화된 이해.
```

**Source 타입:**

| type | 저장 방식 | 비고 |
|---|---|---|
| `url` | URL + 크롤링 텍스트 | 원본 사이트 소멸 대비 텍스트 보존 |
| `pdf` | 파일 저장소(S3) 참조 + 추출 텍스트 | |
| `youtube` | URL + 자막 텍스트(yt-dlp) | 영상 자체는 저장 불가 |
| `text` | 입력 텍스트 그대로 | |
| `conversation` | AI 대화 전체 JSON | AI와 상호작용으로 만든 지식의 출처 |

**레이어 간 연결:**
- Source → Chunk: 1:N (하나의 Source에서 여러 Chunk)
- Chunk → Node: N:M (`node_sources` 브릿지 테이블)
- 하나의 Node는 여러 Source에서 나올 수 있다.
- 사용자가 직접 생성한 Node는 Source 없이도 존재 가능.
- Source 삭제 시 Node는 유지 (soft delete 전략으로 계보 보존).

**노드 추출 흐름:**
```
Source 저장 → Chunk 분할 → 임베딩 생성(비동기) → LLM 노드 추출 제안
  → 사용자 리뷰·확정(필수 게이트) → Node + 관계 확정 저장
```
- LLM 응답은 structured output(JSON schema)으로 강제.
- 확정 전 상태: `draft`. 확정 후: `confirmed`. 사용자가 거부하면 `rejected`.
- 임베딩 생성은 비동기(BackgroundTask). Node 저장은 동기.
- Node 테이블에 `embedding vector(1536)` 컬럼을 Phase 1-A부터 예약.

### 5.5 스코프 — 개인 / 그룹 + 공개 플래그

모든 노드(P/C/M/D)는 **정확히 하나의 Space에 속한다.** Space 타입은 두 가지다.

| 타입 | 설명 | 생성 방식 |
|---|---|---|
| `personal` | 개인 지식 공간. 사용자당 1개. | 회원가입 시 자동 생성 |
| `group` | 그룹 협업 공간. 여러 멤버. | 사용자가 명시적 생성 |

공개 여부는 Space 타입과 독립된 `is_public` 플래그로 제어한다.
- `is_public = false` (기본): 본인(personal) 또는 멤버(group)만 접근.
- `is_public = true`: 외부인이 읽기 가능. 단 가져가기는 항상 이해 게이트(재구성) 경유.

> 이전 버전의 "공유(shared)" 스코프는 제거. `is_public` 플래그로 대체. "공유"는 별도 공간이 아니라 공개된 개인/그룹 공간이다.

**개인 공간 확장성:** 현재는 사용자당 1개로 제한(application 레벨). 스키마는 복수를 지원하므로 추후 "프로젝트별 공간" 기능 추가 시 스키마 변경 없음.

### 5.6 Space 모델 — 소유권과 멤버십

```
users
  id, personal_space_id (기본 개인 공간, 빠른 조회용)

spaces
  id          UUID PK
  type        ENUM('personal', 'group')
  name        VARCHAR
  is_public   BOOLEAN DEFAULT false
  created_by  UUID    -- 생성자 (불변, 감사 목적)
  settings    JSONB   -- 그룹 설정 (승인 정책 등)
  created_at  TIMESTAMPTZ
  deleted_at  TIMESTAMPTZ

space_members
  space_id  UUID
  user_id   UUID
  role      ENUM('owner', 'admin', 'member', 'viewer')
  joined_at TIMESTAMPTZ
  PK: (space_id, user_id)
```

**역할 정의:**

| 역할 | 권한 |
|---|---|
| `owner` | 공간 삭제, 모든 권한. Space는 owner에 귀속. 변경 가능(양도). |
| `admin` | 멤버 초대/제거/역할 변경. 노드 생성·편집·삭제. |
| `member` | 노드 생성·편집 (승인 정책에 따라 직접 반영 또는 제안). |
| `viewer` | 읽기만 가능. |

**소유권 구조:**
- 개인 공간(`personal`): `space_members`에 본인이 `role='owner'`로 등록.
- 그룹 공간(`group`): `space_members`의 `role='owner'`가 그룹 소유자. `spaces` 테이블에 `owner_id` 컬럼 없음.
- owner 변경: 기존 owner → 다른 역할 강등 + 신규 owner 승격 (application 레벨 트랜잭션).
- owner는 space당 정확히 1명 (application 레벨 보장).
- 권한 체크 로직은 공간 타입 분기 없이 `space_members.role`로 통일.

### 5.7 그룹 노드 변경 승인 (Node Proposal)

그룹 공간에서 `member`가 노드를 수정하는 것은 "main에 직접 push"와 같다. 이를 선택적으로 제어하는 승인 게이트를 제공한다.

```
spaces.settings JSONB:
{
  "require_approval": false,         -- false: 즉시 반영 (기본)
  "auto_approve_roles": ["owner", "admin"]  -- 이 역할은 승인 없이 즉시 반영
}
```

`require_approval = true`일 때 member의 노드 변경은 `node_proposals` 테이블에 제안으로 기록되고, owner/admin이 승인 후 실제 반영된다.

```
node_proposals
  id            UUID PK
  space_id      UUID
  proposed_by   UUID
  node_id       UUID NULL    -- null이면 새 노드 생성 제안
  action        ENUM('create', 'update', 'delete')
  payload       JSONB        -- 제안 노드 데이터
  status        ENUM('pending', 'approved', 'rejected')
  reviewed_by   UUID NULL
  reviewed_at   TIMESTAMPTZ NULL
  created_at    TIMESTAMPTZ
```

### 5.8 공간 간 지식 이동 (지식 융합의 핵심 메커니즘)

다른 공간의 지식을 가져오는 것은 **복사·참조가 아니라 재구성**이다. 원본을 그대로 가져오는 것이 아니라, 원본을 보고 대상 공간의 맥락에 맞게 새로 작성하는 것이다. 이동은 다음 게이트를 반드시 거친다.

```
원본 노드(공개된 개인/그룹 공간)
  → AI 번역 제안 (대상 공간의 맥락으로 변환, 보조)
  → 사용자 이해·정리 (필수 게이트: 직접 수정·재구성)
  → 대상 공간에 새 노드로 확정 저장
```

- **이동 경로**: 개인 ↔ 그룹, 공개 공간 → 개인/그룹 (단방향 읽기 후 재구성).
- **결과 노드**는 대상 공간의 언어/맥락으로 유지된다. 원본과 동일한 내용이어도 별개의 노드.
- **출처(provenance)**: 새 노드는 원본과 `node_lineage` 테이블로 계보 연결. 5.2의 6종 의미 관계와 분리된 축.
  - Phase 2: `원본 node_id + 이동 시각 + 이동 유저` 최소 기록.
  - Phase 3: 사용자가 어떻게 수정했는지 diff 저장 추가.
- **근거**: 원본을 그대로 참조하면 1.2의 블랙박스 문제가 재발한다. 재구성 게이트가 Epistruct를 LLM Wiki와 가르는 지점이다.

## 6. 기능 요구사항 (Phase별)

### Phase 1 — 개인 지식 그래프 (내 두뇌를 외부화)

**1-A. 저장·구조화 (먼저 동작 가능한 서비스)**
- F0. 인증: 회원가입 / 로그인 / 세션 관리. *(Phase 1-A 착수 전제 조건)*
- F1. 혼합 입력: URL / PDF / 유튜브 / 텍스트 붙여넣기.
  - 각 입력 타입은 Source로 저장(5.4 데이터 계층 모델).
  - URL: 크롤링 텍스트 보존. YouTube: yt-dlp 자막 추출. PDF: 파일 저장소 업로드.
- F2. 입력 소스에서 노드(C/M) + 관계 자동 추출 (사용자 확정 전제).
  - Source → Chunk 분할 → LLM structured output으로 노드 후보 추출.
  - 추출 결과는 `draft` 상태로 리뷰 화면에 제공. 사용자가 수정·확정(`confirmed`) 또는 거부(`rejected`).
  - 임베딩 생성은 비동기. Node 테이블에 `embedding vector(1536)` 컬럼 Phase 1-A부터 예약.
- F3. 지식 그래프 시각화 및 인터랙티브 탐색.
  - MVP: **웹 우선**. 네이티브 그래프는 Phase 1-B로 이월. 네이티브 MVP는 리스트/트리 뷰 대체.
- F4. 노드 직접 편집/병합/삭제 — 사람의 개입·통제 보장 (1.2 결함의 직접 해법).
  - 삭제는 soft delete(`deleted_at`) 기본. 물리 삭제 없음.

**1-B. 지능화 (차별화 핵심)**
- F5. 온보딩: 학습 스타일·관심 도메인 파악 (샘플 제시형 + 실제 해보기형 혼합).
- F6. 맥락 기반 능동 제안: "지금 이 맥락에서 알아야 할 것" 제시.
- F7. 컨텍스트 압축 + 중복 감지: 임베딩 유사도로 유사·중복 노드 통합.
- F8. 원리 추출(P): 노드에서 표면 사실과 분리된 원리 레이어를 추출(5.3 진입 기준 적용).
- F9. 원리 전이 제안: 도메인이 달라도 원리가 유사한 노드를 `ANALOGOUS_TO`로 연결해 통찰 제안.
- F10. 학습 상호작용: AI가 답을 주는 대신 정리 방식을 질문·제안(소크라테스식).

### Phase 2 — 그룹 + 지식 융합 (다른 두뇌와 연결)

- F11. 공개 설정: Space 단위 `is_public` 토글. 공개된 공간의 노드는 외부인이 열람 가능.
- F12. 공간 간 이동(가져오기): 공개된 개인/그룹 공간의 노드를 내 공간으로 재구성. 항상 5.8의 이해 게이트 경유.
- F13. AI 지식 번역: 원본 노드를 대상 공간의 맥락으로 변환 제안 (게이트의 보조 단계).
- F14. 충돌·중복 감지 + 병합 제안: 가져온 노드가 대상 공간의 기존 그래프와 중복·상충할 때.
- F15. 출처(provenance) 추적: 이동된 노드의 계보 기록 (`node_lineage`). Phase 3 기여도·격차의 기반.
- F16. 그룹 공간 운영: 그룹 생성·멤버 초대(역할 포함)·노드 공동 관리·승인 정책 설정(5.7 Node Proposal).

### Phase 3 — 집단 지성 (커뮤니티가 함께 성장)

- F17. 집단 지식 그래프: 같은 개념을 사람들이 어떻게 다르게 연결했는지 시각화.
- F18. 지식 격차 발견: "이 연결고리가 너에게는 없다" 알림.
- F19. 학습 경로 추천: 유사 지식 구조 사용자가 다음에 배운 것 기반 추천.
- F20. 원리 전이 커뮤니티 + 기여도 측정: 도메인 간 원리 연결을 집단이 함께 발견·검증하고, 출처 계보로 기여를 추적.

## 7. 대표 사용자 시나리오

**S1. 첫 입력 (온보딩 직후)**
사용자가 "SQL 인덱스" 아티클 URL을 붙여넣는다 → AI가 핵심 개념 노드를 뽑고, 온보딩에서 파악한 스타일(예: 사례형)에 맞춰 정리 방식을 제안한다 → 사용자가 수정·확정하면 그래프에 반영된다. (확정은 항상 사람이 한다.)

**S2. 능동 꺼내주기 + 원리 전이**
사용자가 "쿼리 튜닝"을 입력하는 순간, AI가 연관 노드(인덱스, 실행계획)를 끌어온다. 나아가 "이 캐싱 원리는 네가 전에 정리한 `C:cpu-cache-locality`와 같은 `P:time-space-tradeoff`다"라고 도메인을 넘는 연결을 제안한다.

**S3. 공간 간 지식 융합**
동료의 그룹 공간(is_public=true)에 올라온 `C:transaction-isolation` 노드를 사용자가 자기 개인 공간으로 가져온다 → AI가 사용자의 기존 "동시성" 맥락에 맞게 번역 제안 → 사용자가 직접 수정·재구성해 새 노드로 확정한다. 원본과는 `node_lineage`로 계보가 연결되며, 사용자의 그래프는 항상 사용자의 언어로 유지된다. 원본 노드는 변경되지 않는다.

## 8. 기술 아키텍처 (확정)

### 8.1 제약 조건
- 비용: 무료~저비용 수준에서 운영 가능해야 한다.
- 플랫폼: iOS / Android / Web 동시 지원.
- DB: PostgreSQL (확정).

### 8.2 클라이언트 — Expo (React Native + RN Web) [확정]
- 한 코드베이스(RN 컴포넌트: `View`, `Text` 등)로 iOS / Android / Web을 동시 커버.
- 일반 React 웹과 달리 모바일·웹 코드가 분리되지 않는다 (RN Web이 RN 컴포넌트를 웹 DOM으로 렌더링).
- Expo가 RN + RN Web 빌드를 통합 관리. 무료. "멀티플랫폼 + 무료" 조건 충족.
- 그래프 시각화: 웹은 react-force-graph / D3, 네이티브는 Skia 기반 렌더 검토.

### 8.3 인증 — Supabase Auth [확정]

**선택 근거:** 인증 구현보다 지식 그래프 핵심 기능에 집중. 보안 기본기(Refresh token rotation, 이메일 인증 등) 검증된 구현 제공. 무료 50,000 MAU. JWT 표준 준수로 추후 자체 구현 전환 가능.

**아키텍처:**
```
Expo 클라이언트
  → Supabase Auth SDK로 로그인 (소셜/이메일)
  → Supabase가 JWT 발급
  → FastAPI 요청 시 Authorization 헤더에 JWT 포함
  → FastAPI가 Supabase JWKS endpoint로 토큰 검증
  → 검증된 user_id로 내 PostgreSQL 조회
```

**내 DB와의 연동:**
- Supabase Auth가 사용자 인증 데이터를 관리 (`auth.users`).
- 내 PostgreSQL에 `users` 테이블 별도 유지 (프로필, `personal_space_id`, 학습 스타일 등).
- Supabase webhook → 가입 시 내 `users` 테이블에 row 자동 생성.
- 내 코드에서 사용자 식별은 Supabase가 발급한 `user_id`(UUID)로 통일.

**모듈 구조에서 auth 모듈의 역할:**
- JWT 검증 미들웨어 (Supabase JWKS 사용).
- `users` 테이블 CRUD (프로필, 설정).
- Supabase webhook 수신 엔드포인트.
- 인증 자체(로그인/가입/토큰 발급)는 Supabase가 처리. auth 모듈은 검증과 프로필 관리만 담당.

### 8.4 백엔드 — Modular Monolith (FastAPI) [확정]

**아키텍처 전략: Modular Monolith → MSA 점진 전환**

초기에는 단일 FastAPI 서버로 시작하되, 내부를 서비스 경계에 따라 모듈로 분리한다. 모듈 경계를 지키면 필요 시 서비스로 뽑아낼 수 있다.

```
epistruct-api/
├── auth/           JWT 검증 + users 프로필 모듈
├── knowledge/      노드+관계+그래프 모듈 (핵심 도메인)
├── space/          공간+멤버십 모듈
└── ai_pipeline/    LLM+임베딩+추출 모듈
```

**모듈 경계 규칙:**
- 각 모듈은 자기 테이블만 직접 접근한다. 타 모듈 테이블 직접 JOIN 금지.
- 모듈 간 호출은 반드시 service layer를 통한다.
- DB FK 없음. 참조 무결성은 애플리케이션 레벨에서 보장.
- 모든 레코드는 soft delete(`deleted_at`) 기본. 물리 삭제 없음.

**MSA 분리 트리거 (사전 정의):**
- `ai_pipeline`: LLM 비용이 전체 운영비 30% 초과, 또는 응답 지연이 핵심 UX에 영향 시.
- `knowledge`: DAU 10,000 초과, 또는 팀이 도메인별로 분리될 때.

**API 분리 원칙:**
- `/api/v1/` — 외부 연계용 공개 API (breaking change 불가, 버전 관리).
- `/internal/` — 모듈 간 내부 호출용 (자유롭게 변경 가능).

AI/LLM 생태계(LangChain·LlamaIndex·임베딩 파이프라인) 1급 지원. 비동기 처리·LLM 스트리밍에 적합. 사용자의 Python 학습 목표와 시너지.

### 8.5 데이터 — PostgreSQL + 확장 [확정]

**테이블 구조:**
- `users`: 사용자. `personal_space_id` 컬럼으로 기본 개인 공간 참조.
- `spaces`: 공간 테이블. `type(personal/group)`, `is_public`, `settings JSONB`. Phase 1-A는 personal만 구현하되 스키마 전체 포함.
- `space_members`: 공간 멤버십. `role(owner/admin/member/viewer)`. 개인 공간도 본인이 owner로 등록.
- `node_proposals`: 그룹 노드 변경 제안 (require_approval=true 시 사용). Phase 2.
- `nodes`: 단일 테이블 + `node_type` enum(P/C/M/D). `space_id`, `created_by` 포함.
- `node_relationships`: 6종 관계를 인접 리스트로 표현. 재귀 CTE로 그래프 탐색.
- `sources`: 원본 입력 저장 (`type`, `raw_content`, `metadata`).
- `chunks`: Source 분할 가공본 + `embedding vector(1536)`.
- `node_sources`: Node ↔ Source N:M 브릿지 (출처 추적 기반).
- `node_lineage`: 공간 간 이동 계보 (6종 의미 관계와 분리된 별도 테이블). Phase 2.

**설계 원칙:**
- DB FK 없음. `space_id`, `user_id` 등 참조 컬럼은 UUID만 보유. 검증은 service layer.
- Soft delete: 모든 주요 테이블에 `deleted_at TIMESTAMPTZ` 컬럼.
- 임베딩: Chunk 테이블 + Node 테이블 모두 `embedding vector(1536)` 예약. 생성은 비동기.
- 이벤트 일관성: 상태 변경은 Outbox Pattern으로 이벤트 유실 방지 (MSA 전환 시 기반).

**pgvector 확장**: 의미 기반 유사도 검색. 중복 감지(F7), 능동 제안(F6), 원리 전이(F9)의 공통 기반.

(향후 옵션) 그래프 탐색이 본격화되면 Apache AGE(Cypher) 검토.

### 8.6 AI 레이어
- 외부 LLM(Claude API 등) + RAG(pgvector 검색 결합).
- 노드 추출: structured output(JSON schema) 강제. 응답 스키마: `{node_type, label, description, relationships[]}`.
- P 노드 생성: AI 제안 + 사용자 확정 게이트 필수. 자동 생성 없음.
- 원리 추출/전이, 스코프 간 번역은 별도 프롬프트 파이프라인으로 모듈화.
- 비용 통제: 캐싱·배치·임베딩 선검색으로 LLM 호출 최소화.

## 9. 비기능 요구사항

- **동시성**: 그룹 공간 노드는 여러 멤버가 동시에 편집할 수 있다 → 충돌을 안전하게 해결(낙관적 락 / 버전 관리 / 병합 제안). `require_approval=true` 시 Node Proposal로 충돌 자체를 예방.
- **편의성**: 입력은 최소 마찰(붙여넣기 한 번), 정리는 사용자 주도이되 AI가 부담을 덜어준다.
- **인지 가능성**: 그래프가 커져도 사람이 전체를 파악·통제할 수 있어야 한다(압축·계층화·뷰 분리).
- **성능**: 그래프가 커져도 탐색·제안 응답이 일정 수준 유지.
- **보안/프라이버시**: 노드 단위 스코프, 출처 구분, 개인 데이터 격리.
- **비용 통제**: LLM 호출 효율화로 운영비 최소화.

## 10. 성공 지표 (초안)

- 활성화: 온보딩 완료율, 첫 입력까지 도달률.

## 11. 리스크와 오픈 이슈
